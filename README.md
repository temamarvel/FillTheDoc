# либ FillTheDoc

`FillTheDoc` — macOS-приложение для полуавтоматического заполнения договоров и шаблонов в формате `DOCX` на основе реквизитов компании.

Приложение помогает пройти весь сценарий в одном окне:
- загрузить шаблон `DOCX`;
- загрузить файл с реквизитами компании (`DOC`, `DOCX`, `PDF`, `XLS`, `XLSX`, `TXT`);
- извлечь текст из файла;
- распознать реквизиты через LLM;
- проверить и при необходимости исправить поля вручную;
- заполнить шаблон и сохранить новый `DOCX`;
- дополнительно получить строку для вставки в Google Sheets.

## Для чего это нужно

`FillTheDoc` полезен, если вы регулярно:
- заполняете типовые договоры, анкеты и шаблоны;
- получаете реквизиты клиентов в разных форматах;
- хотите сократить ручной copy/paste;
- хотите оставить человеку финальную проверку перед выгрузкой документа.

## Что умеет приложение

- Работает как нативное приложение для macOS на `SwiftUI`.
- Принимает шаблон только в формате `DOCX`.
- Извлекает текст из файлов реквизитов в форматах:
  - `DOC`
  - `DOCX`
  - `PDF`
  - `XLS`
  - `XLSX`
  - `TXT`
- Использует OpenAI-модель `gpt-4o-mini` для преобразования извлечённого текста.
- Даёт пользователю форму подтверждения и редактирования данных перед заполнением шаблона.
- Валидирует поля локально и частично сверяет их с DaData.
- Поддерживает заполнение плейсхолдеров в `DOCX`, включая основные XML-части документа.
- Экспортирует готовый документ через стандартное окно сохранения macOS.
- Формирует TSV-строку для вставки в Google Sheets и копирует её в буфер обмена.

## Как это работает

Базовый сценарий такой:

1. Вы перетаскиваете в приложение шаблон `DOCX`.
2. Вы перетаскиваете файл с реквизитами компании.
3. Приложение извлекает из него текст локально на Mac.
4. Извлечённый текст отправляется в OpenAI для структурирования в поля компании.
5. Вы проверяете форму, исправляете ошибки и нажимаете **«Применить»**.
6. После подтверждения данных становится доступна кнопка **«Извлечь реквизиты и заполнить шаблон»**.
7. Приложение подставляет значения в плейсхолдеры шаблона и предлагает сохранить новый `DOCX`.
8. Если данные подтверждены, дополнительно формируется строка для Google Sheets.

## Архитектура проекта

Ниже не просто список папок, а карта ответственности слоёв. Если вы впервые открыли код, лучше читать проект именно в таком порядке.

### 1. Точка входа и orchestration UI

- `FillTheDoc/FillTheDocApp.swift` — точка входа приложения.
- `FillTheDoc/Views/MainView/MainView.swift` — корневой экран, который собирает основные блоки интерфейса.
- `FillTheDoc/Views/MainView/MainViewModel.swift` — главный координатор пользовательского сценария.

`MainViewModel` — это центральный orchestration-слой приложения. Он **не** реализует сам extraction, валидацию, работу с Keychain или DOCX, а только:
- принимает события от UI;
- вызывает специализированные сервисы;
- хранит observable state;
- следит, чтобы шаги сценария происходили в правильном порядке.

Именно поэтому большая часть важной архитектуры читается не из view, а из `MainViewModel` и его зависимостей.

### 2. Извлечение текста из входных файлов

Папки:
- `FillTheDoc/TextExtraction/`
- `FillTheDoc/Extractors/`

Здесь есть два уровня:

1. `DocumentTextExtractorService` — верхнеуровневый фасад extraction-пайплайна.
2. Конкретные адаптеры (`PlainTextExtractor`, `PDFKitTextExtractor`, `TextutilOfficeExtractor`) — низкоуровневые механизмы для отдельных форматов.

Почему сделано именно так:
- UI и orchestration не должны знать, чем отличается `PDFKit` от `textutil`;
- конкретные extractors проще тестировать и заменять по отдельности;
- фасад может централизованно решать sandbox-вопросы, нормализацию текста и сбор диагностики.

### 3. Граница между LLM и внутренней моделью

Папка: `FillTheDoc/DataModels/`

- `CompanyDetails` — каноническая структурированная модель реквизитов.
- `LLMExtractable` — контракт для типов, которые можно просить у модели в виде JSON.
- `LegalForm` — типизированный справочник правовых форм и их нормализация.

Это важная граница проекта:
- LLM возвращает **черновик** в `CompanyDetails`;
- пользователь затем подтверждает или исправляет этот черновик;
- только после подтверждения данные считаются источником истины для заполнения шаблона.

### 4. Placeholder-domain

Папка: `FillTheDoc/Placeholder/`

Это слой, который объясняет приложению, **какие ключи вообще существуют**, как их показывать, валидировать и вычислять.

Ключевые типы:
- `PlaceholderKey` — identity плейсхолдера;
- `PlaceholderDescriptor` — метаданные для UI и библиотеки плейсхолдеров;
- `PlaceholderRegistry` — единый реестр знаний о плейсхолдерах;
- `PlaceholderResolutionContext` — контекст, из которого вычисляются значения;
- `TemplatePlaceholderResolver` — сборщик итогового словаря значений;
- `CompanyDetailsAssembler` — mapper между `CompanyDetails` и placeholder-словарями.

Почему это отдельный слой, а не логика прямо во view model:
- UI, валидация, библиотека плейсхолдеров и DOCX fill должны опираться на одни и те же определения;
- иначе список ключей, правила нормализации и derived-поля начали бы расходиться по проекту.

### 5. Подтверждение данных пользователем

Папки:
- `FillTheDoc/Views/DocumentDataView/`
- `FillTheDoc/Validation/`

После ответа модели пользователь видит форму подтверждения.

Это отдельный этап по смыслу:
- до нажатия **«Применить»** данные считаются только предварительным черновиком;
- после нажатия **«Применить»** формируется финальный набор значений для шаблона;
- внешняя сверка с DaData даёт подсказки, но не переписывает значения автоматически.

Такой компромисс выбран специально: приложение экономит ручной труд, но оставляет человеку финальное решение в юридически чувствительном сценарии.

### 6. Заполнение DOCX

Основной orchestration живёт в `MainViewModel`, а техническая работа с шаблоном делегируется зависимостям из `DocxUtils` и helper-типам проекта.

Пайплайн такой:

1. из формы получается словарь `[PlaceholderKey: String]`;
2. словарь переводится в string-keyed формат для шаблонизатора;
3. условные блоки шаблона сначала собираются/раскрываются;
4. затем выполняется обычная подстановка значений;
5. результат экспортируется через системный `fileExporter`.

Порядок «сначала условные блоки, потом обычная замена» важен: control tokens не являются обычными пользовательскими плейсхолдерами и должны обрабатываться отдельным этапом.

### 7. Инфраструктурные сервисы

- `Keychain/` — безопасное хранение OpenAI API key.
- `AppUpdate/` — проверка новых версий через GitHub Releases.
- `Views/CopyString/` — формирование TSV-строки для внешнего процесса работы с Google Sheets.

Это вспомогательные подсистемы. Они не участвуют в расчёте плейсхолдеров напрямую, но закрывают важные реальные сценарии приложения.

## Ключевые архитектурные компромиссы

- **Human-in-the-loop вместо полного автомата.** LLM ускоряет извлечение, но не заменяет ручное подтверждение.
- **macOS-first extraction.** Для office-файлов используется системный `textutil`, потому что это упрощает стек зависимостей, но делает поведение привязанным к macOS.
- **Soft-fail в extraction.** Если текст извлечь не удалось, pipeline старается вернуть диагностируемый пустой результат, а не мгновенно ронять весь сценарий.
- **Единый placeholder-registry.** Это немного увеличивает число типов, но резко снижает риск рассинхронизации между UI, валидацией и export.
- **После `Применить` форма становится источником истины.** Не исходный LLM JSON, а именно подтверждённые пользователем значения идут в шаблон и в строку для Google Sheets.


## Использование

### Шаг 1. Загрузите шаблон

Левая зона принимает только шаблон `DOCX`:
- заголовок в UI: **«Шаблон (DOCX)»**.

Если шаблон существует и доступен, зона считается валидной.

### Шаг 2. Загрузите реквизиты

Правая зона принимает файл с реквизитами:
- `DOC`
- `DOCX`
- `PDF`
- `XLS`
- `XLSX`
- `TXT`

После загрузки приложение автоматически запускает извлечение текста и распознавание реквизитов.

### Шаг 3. Подтвердите данные

После распознавания появляется форма, где можно:
- проверить поля компании;
- вручную исправить ошибки;
- ввести:
  - `Комиссия, %`
  - `Мин. комиссия, руб`.

Поля валидируются локально. Для части реквизитов дополнительно выполняется сверка с DaData после потери фокуса.

### Шаг 4. Заполните шаблон

Кнопка **«Извлечь реквизиты и заполнить шаблон»** активируется только если:
- шаблон существует;
- файл реквизитов существует;
- есть сохранённый OpenAI API key;
- пользователь нажал **«Применить»** и подтвердил данные формы.

После нажатия приложение:
- создаёт временный заполненный `DOCX`;
- открывает системный экспортёр macOS;
- предлагает имя вида `<template>_filled.docx`;
- формирует строку для Google Sheets и копирует её в буфер обмена.

## Поддерживаемые входные форматы

| Тип файла | Поддержка | Как обрабатывается |
| --- | --- | --- |
| `DOCX` | Да | Через системный office extractor |
| `DOC` | Да | Через системный office extractor |
| `PDF` | Да | Через `PDFKit` |
| `XLS` | Да | Через системный office extractor |
| `XLSX` | Да | Через системный office extractor |
| `TXT` | Да | Как plain text |
| Другие расширения | Нет | Возвращается ошибка неподдерживаемого формата |

### Важные ограничения

- Для `PDF` приложение умеет извлекать текст, но не делает OCR.
- Если PDF состоит только из сканов/картинок, результат может быть пустым.
- Извлечённый текст нормализуется и обрезается до `60_000` символов.
- Для `DOC` / `DOCX` / `XLS` / `XLSX` используется внешний механизм текстового извлечения на базе системных утилит macOS.

## Поддерживаемые поля компании

LLM заполняет структуру `CompanyDetails` со следующими полями:

| Ключ | Что означает |
| --- | --- |
| `company_name` | Название компании |
| `legal_form` | Правовая форма (`ООО`, `АО`, `ИП` и т.д.) |
| `ceo_full_name` | ФИО руководителя |
| `ceo_full_genitive_name` | ФИО руководителя в родительном падеже |
| `ceo_shorten_name` | Краткое ФИО руководителя |
| `ogrn` | ОГРН / ОГРНИП |
| `inn` | ИНН |
| `kpp` | КПП |
| `email` | Email |
| `address` | Юридический адрес |
| `phone` | Телефон |

Дополнительно в итоговый словарь для шаблона добавляются вычисляемые поля:

| Ключ | Источник |
| --- | --- |
| `fee` | Вводится пользователем |
| `min_fee` | Вводится пользователем |

| `date` | Текущая дата в формате `«dd» MMMM yyyy г.` |
| `ceo_role` | `Генеральный директор` или `Индивидуальный предприниматель` |
| `rules` | `Устава` или текст про запись в ЕГРИП |

## Плейсхолдеры в шаблоне DOCX

Приложение ищет и заменяет плейсхолдеры в формате:

```text
<!company_name!>
<!inn!>
<!full_company_name!>
<!date!>
```

### Пример

Если в шаблоне есть текст:

```text
Исполнитель: <!full_company_name!>, ИНН <!inn!>
```

то после заполнения он будет заменён значениями из формы.

### Что важно учитывать

- Если в шаблоне есть ключ, которого нет в словаре значений, по умолчанию плейсхолдер **сохраняется как есть**.
- Замена работает по XML-частям `DOCX`, а не по визуальному тексту из Word.
- Сканер и replacer умеют собирать плейсхолдеры даже если текст разбит на несколько runs внутри абзаца.
- Обрабатываются стандартные части документа; поддержка сносок, endnotes и comments включена.

## Валидация данных

### Локальная валидация

Форма проверяет:
- непустое название компании;
- ИНН;
- КПП;
- ОГРН / ОГРНИП;
- ФИО;
- краткое ФИО;
- email;
- адрес;
- телефон;
- правовую форму;
- проценты комиссии.

### Сверка с DaData

После ввода или правки ключевых реквизитов приложение может дополнительно показать предупреждения, если данные слабо совпадают с DaData, например:
- ИНН не совпадает;
- КПП не совпадает;
- ОГРН не совпадает;
- название компании слабо похоже;
- ФИО руководителя слабо похоже;
- адрес слабо похож.

Это именно предупреждения для оператора, а не полностью автоматическое исправление данных.

## Google Sheets

После успешной подготовки документа приложение формирует TSV-строку и копирует её в буфер обмена.

Сейчас в строку входят:
- наименование компании;
- ФИО руководителя;
- ИНН;
- телефон;
- email;
- дата договора;
- комиссия;
- минимальная комиссия;
- несколько пустых колонок-заготовок.

Если строка сформирована, в интерфейсе может отображаться блок с готовым содержимым для вставки в Google Sheets.


## Ограничения текущей версии

- Приложение работает только с шаблонами `DOCX`.
- OCR для сканированных PDF пока не реализован.
- Ошибки в UI в основном логируются в консоль и не всегда выводятся пользователю как отдельные уведомления.
- Подстановка значений зависит от того, насколько предсказуемо оформлен шаблон Word.
- Качество распознавания зависит от качества исходного текста и ответа модели.
- Список поддерживаемых плейсхолдеров жёстко связан с `CompanyDetails` и `DocumentData`.

## Troubleshooting

### Кнопка заполнения неактивна

Проверьте, что:
- выбран шаблон `DOCX`;
- выбран файл с реквизитами;
- сохранён OpenAI API key;
- в форме нажата кнопка **«Применить»**;
- в форме нет блокирующих ошибок валидации.

### После загрузки файла реквизитов ничего не появилось

Возможные причины:
- файл содержит слишком мало читаемого текста;
- PDF состоит из сканов без текстового слоя;
- внешний extractor не смог обработать office-файл;
- OpenAI-запрос завершился ошибкой;
- ошибка выведена только в консоль Xcode.

### Плейсхолдеры не заменились

Проверьте, что:
- шаблон действительно в формате `DOCX`;
- плейсхолдер записан как `<!key!>`;
- ключ существует в словаре значений;
- вы используете правильное имя, например `company_name`, `inn`, `date`, `full_company_name`.

### API key запрашивается снова

Проверьте:
- был ли ключ сохранён без ошибок;
- доступно ли сохранение в `Keychain` на этом Mac;
- нет ли ошибок в консоли при загрузке ключа.
